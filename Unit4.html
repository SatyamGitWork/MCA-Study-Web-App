1. Difference between `find` and `findOne`

This section breaks down the two primary methods for querying documents in MongoDB. The key difference lies in what they return: a cursor for multiple documents versus a single document object.

`db.collection.find(query, projection)`

  • Purpose: Retrieves multiple documents that match the query criteria.
  • Return Value: Returns a cursor, which is a pointer to the result set. You then iterate over the cursor to access the documents. Even if only one document matches, it returns a cursor.
  • Example: db.users.find({ status: "active" }) would return a cursor pointing to all users with an "active" status.

`db.collection.findOne(query, projection)`

  • Purpose: Retrieves only the first document that matches the query criteria.
  • Return Value: Returns a single document (as a JSON-like object), not a cursor. If no document matches, it returns null.
  • Example: db.users.findOne({ userId: 123 }) would return the single document for the user with userId 123.

2. Difference between ORM and ODM

Understanding the difference between Object-Relational Mapping (ORM) and Object-Document Mapping (ODM) is essential for choosing the right library to connect your application to a database. The choice depends entirely on whether you're using a SQL (Relational) or NoSQL (Document) database.

ORM (Object-Relational Mapping)

  • Used with: Relational Databases (SQL) like MySQL, PostgreSQL, etc.
  • Concept: Maps data from tables and rows to objects and classes in your code.
  • Example Libraries: Sequelize (Node.js), SQLAlchemy (Python), Hibernate (Java).

ODM (Object-Document Mapping)

  • Used with: Document Databases (NoSQL) like MongoDB.
  • Concept: Maps data from collections and documents to objects and classes in your code.
  • Example Libraries: Mongoose (Node.js), MongoEngine (Python).

Data Model Tables, Rows, Columns Collections, Documents, Fields
Mapping Maps tables/rows to objects/classes Maps collections/documents to objects/classes

3. Why we use NoSQL

NoSQL ("Not Only SQL") databases emerged to solve the limitations of traditional SQL databases, especially for modern, large-scale applications. Here are the primary drivers for adopting NoSQL solutions.

  1. Flexible Data Models: NoSQL databases don't require a predefined schema. This is ideal for unstructured or semi-structured data and allows for rapid development and iteration.
  2. Horizontal Scalability (Scaling Out): They are designed to scale horizontally by adding more servers, which is often more cost-effective than upgrading a single large server (vertical scaling).
  3. High Performance: They are often optimized for specific data models and access patterns (like fast key-value lookups), leading to better performance for certain use cases.
  4. Big Data: Well-suited for handling massive volumes of data and high-velocity read/write operations (e.g., IoT data, social media feeds).
  5. High Availability: Many NoSQL databases have built-in support for replication and distribution, ensuring the system remains available even if servers fail.

4. Why do we use MongoDB?

Among NoSQL databases, MongoDB is exceptionally popular. It hits a "sweet spot" by providing the flexibility of NoSQL with the rich query capabilities often associated with SQL databases. Here's why it's a common choice.

  1. Document-Oriented: It stores data in flexible, JSON-like (BSON) documents. This allows for complex nested structures and arrays in a single record, mapping directly to application objects.
  2. Schema Flexibility: You can change the structure of documents over time without complex database migrations.
  3. Rich Query Language: Provides a powerful query language, including ad-hoc queries, indexing, and aggregations.
  4. Scalability & Availability: It has strong, built-in features for horizontal scaling (Sharding) and high availability (Replication).
  5. Performance: Features like indexing, in-memory computing, and the flexible data model contribute to high performance.

5. List and explain various features of MongoDB.

MongoDB is more than just a simple document store; it's a full-featured database with a rich ecosystem. These core features are what make it a powerful choice for modern applications.

  1. Document-Oriented (BSON): Stores data in BSON (Binary JSON) documents, supporting nested objects and arrays.
  2. Ad-hoc Queries: Supports a rich query language to search by field, range, regular expressions, and more.
  3. Indexing: Supports various types of indexes (single field, compound, text, geospatial) to dramatically improve query performance.
  4. Replication (High Availability): Uses replica sets to provide high availability. A primary node accepts writes, which are replicated to secondary nodes. If the primary fails, a secondary is automatically elected.
  5. Sharding (Horizontal Scalability): Scales horizontally using sharding, which distributes data across multiple servers (shards). This allows the database to handle massive datasets.
  6. Aggregation Framework: A powerful data processing pipeline for running complex analytics and transformations on your data (e.g., `$match`, `$group`, `$sort`).
  7. GridFS: A specification for storing and retrieving large files (like images or videos) that exceed the 16MB document size limit.

6. Explain how to create users in MongoDB. Also enlist and explain various roles of the users.

MongoDB uses Role-Based Access Control (RBAC) to secure your database. You create users within a specific database and assign them roles that grant permissions. This section shows the command for creating a user and explains the most common roles.

Create User Syntax

db.createUser({
      user: "<username>",
    { role: "<roleName>", db: "<databaseName>" },
    // ... other roles
  ]
});

Create User Example

This example creates a user myAppUser in the admin database who can read and write to the productionDB database.

// 1. Switch to the database where you want to create the user
use admin

// 2. Create the user
db.createUser({
  user: "myAppUser",
  pwd: passwordPrompt(),
  roles: [
    { role: "readWrite", db: "productionDB" }
  ]
});

Various Roles of Users

Database-Specific Roles

  • read: Allows the user to read all non-system collections in the database.
  • readWrite: Allows the user to read and write (insert, update, delete) to all non-system collections.
  • dbAdmin: Provides administrative privileges (e.g., index management) but not user management.
  • userAdmin: Allows the user to create, delete, and manage users within that database.

Cluster-Wide (Admin) Roles

These roles are defined in the admin database and apply cluster-wide.

  • readAnyDatabase: read access to all databases.
  • readWriteAnyDatabase: readWrite access to all databases.
  • userAdminAnyDatabase: userAdmin access to all databases.
  • root: The "superuser" with access to all resources and actions.

7. How will you remove the user in MongoDB? Explain.

You can remove a user with the db.dropUser() command. You must be connected to the database where the user was originally created (often the admin database or a specific service database) to run this command.

Syntax

db.dropUser("<username>");
// 1. Switch to the database where the user is defined
use admin

// 2. Drop the user
db.dropUser("myAppUser");

This command will return true if the user was successfully dropped.

8. Give various data types used in MongoDB.

MongoDB uses BSON (Binary JSON), which extends the standard JSON data types to include additional types useful for database operations. Here are the most common BSON data types.

  • String: UTF-8 string.
  • Integer: 32-bit or 64-bit integer.
  • Double: 64-bit floating-point number.
  • Boolean: true or false.
  • Object: An embedded document.
  • Array: An array of values.
  • ObjectId: A 12-byte unique ID, automatically generated as the _id for documents.
  • Date: Stores the date and time as a 64-bit integer (milliseconds since the Unix epoch).
  • Timestamp: A 64-bit value used internally for replication and sharding.
  • Null: Stores a null value.
  • Regular Expression: Stores a regular expression.
  • Binary Data: Stores binary data (e.g., a small image).
  • JavaScript Code: Stores a string of JavaScript code.

9. Write the MongoDB command to create and drop the database.

This section explains the commands for database creation and deletion. A key concept in MongoDB is that databases are created implicitlyâ€”that is, they aren't actually created until you insert data into them.

Create Database

MongoDB creates a database implicitly. The use command switches to a database, and the database is physically created only when you first insert data (e.g., by inserting a document into a collection).

// 2. Insert data. This action creates the database and the collection. db.myCollection.insertOne({ name: "Test" }) // Now, 'myNewDatabase' officially exists.

Drop Database

The db.dropDatabase() command deletes the *current* database you are using. This operation is irreversible and removes all collections and documents within that database.

// 1. Make sure you are using the correct database
use myNewDatabase

// 2. Run the drop command
db.dropDatabase()

10. Write the steps to connect Node.js with MongoDB

This guide shows how to connect a Node.js application to a MongoDB database using the official mongodb driver from NPM. This is the foundational step for building any Node.js application that relies on MongoDB.

Step 1: Install the MongoDB Driver

Open your terminal in your Node.js project folder and run:

In your JavaScript file (e.g., app.js), use the following code. It imports the MongoClient, defines your connection string (URI), and creates an async function to connect, perform an operation, and then close the connection.

// 1. Import the MongoClient
const { MongoClient } = require('mongodb');

// 2. Define the Connection URI
// For a local server on the default port:
const uri = "mongodb://localhost:27017";
// For MongoDB Atlas, get this string from your cluster's "connect" dialog.

// 3. Create a new MongoClient
const client = new MongoClient(uri);

// 4. Create an async function to connect and run operations
async function run() {
  try {
    // 5. Connect the client to the server
    await client.connect();
    
    // 6. Select the database
    const database = client.db("myNewDatabase"); // Specify your database name
    
    // 7. Select the collection
    const collection = database.collection("myCollection"); // Specify your collection name

    // 8. Perform an operation (e.g., find one document)
    const query = { name: "Test" };
    const doc = await collection.findOne(query);

    console.log("Connected successfully to server");
    console.log("Found document:", doc);

  } catch (err) {
    console.error("An error occurred:", err);
  } finally {
    // 9. Ensure the client will close when you finish/error
    await client.close();
    console.log("Connection closed.");
  }
}

// 10. Call the function
run().catch(console.dir);
